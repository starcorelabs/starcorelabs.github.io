Responsive Email

I landed a short contract working remotely for a small marketing company called DemandLab. They wanted my help setting up responsive emails and landing pages. They use Marketo and Wordpress as their software platforms.

Responsive HTML emails are strange. Built using HTML tables, no head, and limited inline styles. It is the height of inefficient development.

I was given a handful of responsive HTML emails to add content to and prepare for review. As well as a couple of responsive HTML email templates that needed to be setup so the client could use them for an ongoing newsletter.

I learned a lot from these projects. Email clients like Outlook, Google, Yahoo, and Mac Mail vary widely in their HTML email rendering capabilities. What you can create is severely limited. They all seem to be technologically stuck in 2006.

Why haven't any of these companies attempted to update the rendering engines within their clients? Some experts cite security concerns. In truth they don't invest in upgrading because there is little financial gain.

Learning responsive HTML also helped me learn more about marketing which I really enjoyed.

The main take away from building out responsive emails is understanding the tools and technologies being used before starting. Knowing that margins and padding do not work and you will be forced to use HTML table cell spacing and kludge techinques is both helpful and dishearting.

Working with responive HTML emails has given me a new appreciation for modern browers and their capabilities.

Aside from the responsive HTML emails. I was also tasked with setting up landing pages within their Wordpress install. This was much more in my wheel house so it took much less time.

The contract ended abruptly so I didn't save any of the code I wrote to my Github.

HCCMIS Side Projects

I finished the templates for the HCCMIS Redesign. They liked my work so much they extended my contract to work on a few side projects.

They needed my help making pages responsive. Instead of using Bootstrap, which I most familiar with, I was directed to use Skeleton. This is a similar responsive framework. However, it is significantly smaller in terms of CSS code and options. It is a single fluid grid with a handful of helper classes. Everything else you must write yourself from scratch.

The main reason to use responsive frameworks like Bootstrap is to speed up development time. In my opinion, Skeleton works best for smaller landing pages or tiny websites. Anything larger or that requires more functionally should be developed with a more robust responsive framework. In this case the manager told me to use Skeleton for these side projects because he didn't know or like Bootstrap.

I had not heard of or used Skeleton before this. My lack of familiarity slowed me down. It took me a few days to wrap my mind around how to best implement the Skeleton grid. The documentation on the Skeleton website is sparse and makes a lot of unspoken assumptions regarding how it works.

After a bit of trial and error I found it best to build my media queries based on min-widths. This meant starting from mobile-sized displays and working my way up to larger screens. This is also known as "Progressive Enhancement".

These small projects varied widely in scope and depth. Generally I was making the pages responsive and adding extras based on what the projects called for.

Because these projects were very small and based on simple Skeleton grids there was no point in adding the code to my Github.

Wordpress Transfers

While I was working for Celadon Trucking they aquired a handful of smaller trucking companies. These smaller trucking companies all used Wordpress for their websites.

The IT Department was tasked with finding a way to transfer these smaller sites over to the on-site servers because the company wasn't willing to keep paying for the Wordpress hosting.

Since the IT Department was a Microsoft shop the developers had no experience with LAMP Stack, CPanel, or Wordpress. They had assumed they would have to rebuild each site from scratch in ASPX, using C# for the middleware processing code, and connect it to an SQL database. Because that's the only model they knew.

I was told at the timeb that the companies where in the process of being folded into Celadon. Most of these sites needed to be static and would eventually be deleted.

I suggested not wasting months rebuilding sites that would end up getting trashed. I showed the developers how to use the free HTTrack Website Copier tool. To make a static copy of each site.

I spent 30 minutes and made copies of all the sites they needed to convert. They were impressed.

I told them it would only take me a day or two to clean up file structures and convert them to ASPX for hosting on the internal servers.

The lead developer told me I saved them hours of tedious programming and thousands of dollars.

Some of the Wordpress sites were still being used by people needing make updates. So, in those cases I asked the Server Admin to setup a sand boxed install of Wordpress on IIS and to give me Admin access to Wordpress itself.

Once that was setup I exported the site data from the original site and then imported it into our clone. I had to spend a few hours going into the site and setting up all of the widgets to match the orginal site.

Last, I copied the existing user accounts from the original site to the clone and sent emails out to those individuals.

It was a bit more time-consuming than the static copy method. It produced the result the company want while saving them money.

Since I didn't actually write any of the code for the sites. I have not added them to my Github account.

Driver Portal

This was a year long web application for a Celadon Trucking. Its purpose was to create a secure responsive site that allows the drivers to access their payment information through their smart phones. The companies analytics gathered over 6 months proved that the majority of drivers were checking their settlement sheets using iPhones.

The company used a Microsoft Development Back-end consisting of an IIS Server, ASPX Pages, C# middleware, SQL Databases, and an AS400 Database.

I was tasked with taking the graphic designer's page mockups and re-creating them using HTML and CSS. Because the company used Microsoft and .Net this project was considerably more frustrating and time-consuming than it should have been.

Instead of using vanilla HTML5 I had to write ASPX pages. These are similar technologies but the differences in how they operate annoyed me. CSS cannot touch ASPX controls or forms directly. These pages where mostly ASPX grids of data. There do exist ASPX attributes that can be attached to page elements in order to then add CSS classes. In my opinion Microsoft web development in general seems like a kludge configuration that hindered my progress.

I have no programming skill or experience in C# or VB. Frankly, I do not like the Microsoft stack at all. It seems over-engineered for no good reason. I will not work in that kind of environment again.

I prefer working purely on front-end using either JavaScript data binding or PHP for the middleware.

The Senior developer worked closely with the AS400 database programmers to make sure the drivers data was secure and none of the code would break.

For my own securty I did not save any of the C# middleware code or config files. I never had access to the database since I was only working on the Front-End code. I saved only the front-end ASPX code I actually wrote for my portfolio. Without those missing files there is no way to build and run this code. I took the screenshots back while I was working for the company.

However, if you are curious you can view the front-end code on my Github repo.

Ruby Challenge

This was one of my first experiments in building a responsive page using Bootstrap.

Bootstrap is a CSS library used by developer to create web pages that will shrink and grow depending upon the size of the screen used to view them. This allows developer to create one set of web pages for a site and they will work on the majority of smart phones, tablets, laptops, and desktop screens.

Initially I used Bootstrap 2.3 which worked but had some quirks. Half way thorough Bootstrap updated to version 3.0 which was a major code break. So, I had to rewrite the page from scratch. This process was tedious but cemented my understanding of the Bootstrap 3 grid system.

Also, I put together the graphics for the page using Photoshop.
