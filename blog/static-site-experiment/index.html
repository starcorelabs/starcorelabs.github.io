Static Site Experiments

2020-09-01

Welcome Back to the Lab!

In my last update I explained I was working on updating my website to include both my blog and portfolio. To do this I wanted to switch from Hexo.io to 11ty.dev.

I decided to switch because Hexo breaks while developing with it and it lacks the ability to easily support both a blog and a portfolio. So, instead of creating a bunch of custom code to make it work I decided to find a more robust alternative.

I watched this YouTube tutorial to learn how to convert my static HTML theme into an 11ty site. After that I spent a week converting my site over to 11ty.

During development there were a few issues I had to work through:

  • The blog listing page wouldn't show any items. I double checked my "for loop". The solution was to rename the json file so it matched the folder it was located in.
  • I had to create templates for all the different page and post types:
    • postlist.njk
    • post.njk
    • projectlist.njk
    • project.njk
  • All of images were too large so I spent 2 hours using Gimp to resize them. This reduced their dimensions and file size.
  • I found the post and project images didn't work with the template syntax. So I used relative paths instead.
  • Due to the limitations in markdown I had to use HTML img tags so I could assign classes for styling.
  • I used flexbox to fix my skills section to layout the icons correctly.
  • I wanted to create an include for the "about" section. But could not get it working. So I put it in the homepage.njk file.

Last I wanted 11ty to output a fully working static site to the "_site" folder so I could manually deploy it to GitHub Pages. Unfortunately, it seems 11ty was designed with Netlify in mind. After reading through the sales pitch and documentation of Netlify I decided it is not for me.

The whole point of creating a static site is so I can host it for free on GitHub Pages. This site is a blog and a place to showcase my projects. I'm not processing any forms or accepting user data.

I think 11ty is great. But the documentation is lacking and assumes too much pre-existing knowledge on the part of its users. Its definitely not for beginners.

The issue I haven't solve yet is that 11ty requires all CSS, JS, and image paths to start with "/". While the local development server is running the site works flawlessly.

However, opening up the index.html rendered static file in "_site" folder shows a broken site in Chrome.

I spent a couple of days researching to find a solution. I experimented with "pathPrefix" in my eleventy.js config. It adds the ability to render the site in a subfolder. Not remove the starting "/" from the file paths.

I tried using this code 

 {{ '/' | url }}
as a relative path substitute for my CSS, JS, and images.

But it only adds the "/" to the paths in the final output.

I've asked questions on the 11ty GitHub without response. On StackOverFlow I received one answer suggesting that static sites made with 11ty can only be deployed using the for profit company Netlify.

This really irritated me because nothing on the 11ty website suggests this is the case.

I keep thinking its all just JavaScript so there must be a way to program the site to render a working local version. I'm just not skilled enough with JavaScript yet to do it.

Instead of waiting around I decided to copy the rendered output to another folder and manually fix all the file paths so the site works. Yes this really sucks. But, I need my site up and running now.

So, I'm currently stuck until I either figure out how to fix this glaring issue that other static site generators don't have. Or spend another week learning and switching to something else.

Going forward I will check to make sure whatever software I choose can output the files I need before I invest my time an energy building with it.

One thing I won't do is use Netlify because I will not give up ownership of my website to a for-profit company.

Update 2020-09-06: Eleventy Works!

I sent the last few days trying to figure out how to set 11ty to generate relative links. I was under the impression that if it didn't work locally it wouldn't work on GitHub Pages. Sometimes its good to be wrong.

On the 11ty GitHub repo under issues I asked for help with deploying my 11ty site to Github Pages. Thankfully a few people offer suggestions.

Finally one person cloned my repo and pushed the contents from _site folder to their own username.github.io repo to test it. I was shocked that it actually worked.

I did the same thing in my dev lab. Again I was surprised that it actually works.

So I'm going to stick with 11ty as my static site generator. Now I can focus on adding posts and projects.

Thanks again everyone who helped!
